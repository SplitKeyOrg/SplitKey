# sk-shares — Shamir shares + mnemonic words (format spec)

The contract crate between `Ceremony/` (splits) and `Keyholders/` (combines).
**`Sealer/` never depends on this crate** — the camera holds only public
keys, and the repo layout enforces it: share-handling code cannot even be
linked into `sealerd`.

## What is shared

One **16-byte window secret** per window, per the per-window release model
(`Sealer/docs/02-key-management.md`): shares of the CRK would expose the
whole epoch on first release; shares of a window secret release exactly one
window.

Derivation (small change to today's `sealer-crypto::kdf`, applied when
`ceremony-cli` lands):

```
window_secret = crypto_kdf_derive_from_key(CRK, subkey_id = window_index,
                                            ctx = "SKWINDOW", len = 16)
seed          = BLAKE2b-256(window_secret)
window keypair = crypto_box_seed_keypair(seed)        # X25519, as today
```

**Why 16 bytes, not 32:** X25519 itself offers ~128-bit security, so a
256-bit seed adds nothing against brute force of the curve — but it would
double every booklet line. 128-bit secrets are the same level BIP39's
standard 12-word wallets rely on. The only derivation difference from today
is the 16-byte intermediate + hash step (today the kdf emits the 32-byte
seed directly). This invalidates previously generated manifests — all
throwaway sim/demo state, regenerated by the next ceremony run.

## Shamir scheme

Byte-wise Shamir over GF(2⁸) (AES polynomial `0x11B`) — the same
construction SLIP-39 and `sharks` use:

- For each of the 16 secret bytes independently: a random polynomial of
  degree `t-1` with the secret byte as the constant term.
- Share *x*-coordinates are `1..=n` (one per keyholder, printed in the
  booklet header; `x = 0` is the secret and never a share).
- Combine: Lagrange interpolation at `x = 0` with any `t` distinct shares.
- Information-theoretic: `< t` shares reveal *nothing* (every candidate
  secret remains equally likely).

`t`/`n` are recorded in the manifest (`threshold_t/n`); they are **not**
encoded per-share.

## Word encoding (one booklet line = one window share)

**14 words from the BIP39 English wordlist** (2048 words = 11 bits each;
we use only the wordlist, not BIP39 derivation — every word is uniquely
identified by its first 4 letters, which keeps hand-entry forgiving).

```
payload  = x (1 byte) ‖ y (16 bytes) ‖ check (2 bytes)      = 19 bytes
check    = BLAKE2b-256(community_id ‖ epoch_be16 ‖ window_index_be64
                        ‖ x ‖ y)[0..2]
encoding = payload as a 152-bit big-endian bitstring, + 2 zero padding
           bits → 154 bits → 14 words of 11 bits
```

The checksum **binds the line to its window, epoch, and community** (all
known from the manifest at combine time), so it catches not just typos but
the two realistic ceremony-room mistakes: typing the *wrong line* of the
right booklet, and typing a line from the *wrong community's* booklet.
Decode rejects nonzero padding bits.

Per-line miss probability for a random typo surviving the checksum:
2⁻¹⁶ ≈ 0.0015 % — and the combine step still verifies the reconstructed
key against the manifest pubkey before anything is decrypted (see
`Keyholders/README.md`), so a corrupt share can never produce silent
garbage.

**Why not SLIP-39:** evaluated and declined for v1 — 256-bit mode costs 33
words/line; its per-share group/threshold metadata duplicates what our
manifest already signs; its checksum cannot bind our window index; and the
Rust implementations are unmaintained. The trade-off given up is
third-party tooling compatibility; we compensate with published test
vectors and a spec (this file) sized for independent reimplementation.

## Booklet scale (sanity check)

18-month manifest at 24 h windows ≈ 548 lines × 14 words per booklet
(~25–30 printed pages). Each line is independent — losing or mistyping one
line affects exactly one day of one keyholder's shares. QR codes per line
are a Keyholder-app-phase addition; the format above is what a QR would
contain in text form.

## API sketch

```rust
pub struct Share { pub x: u8, pub y: [u8; 16] }
pub struct LineCtx<'a> { pub community_id: &'a str, pub epoch: u16, pub window: u64 }

pub fn split(secret: &[u8; 16], t: u8, n: u8) -> Vec<Share>;          // ceremony
pub fn combine(shares: &[Share]) -> [u8; 16];                          // keeper
pub fn encode_words(share: &Share, ctx: &LineCtx) -> String;           // 14 words
pub fn decode_words(line: &str, ctx: &LineCtx) -> Result<Share, ShareError>; // checks checksum
```

Randomness and BLAKE2b come from `sealer-crypto` (libsodium). Test vectors
(fixed secret/coefficients → exact word lines) live in `tests/vectors.rs`
and are published with the spec.
